1. Project Overview 
GL Spark is a one-week, sprint-style capstone project for the GlobalLogic Java Track TE 2.0 program, designed 
to simulate real-world software engineering delivery. Teams will design, build, and demonstrate a complete 
microservice-based application within 7 days, following professional engineering standards. 
 
⚡  This is a baselined, end-to-end delivery. Code must be working, tested, and demonstrable on 
Evaluation Day (10 April 2026).

4. User Story Requirements 
Each team must define a minimum of 4 / more User Stories with full acceptance criteria. User stories must be 
documented in the PRD and mapped to implementation. 
 
FORMAT:  As a [role], I want [feature] so that [benefit]. 
 
4.1 Acceptance Criteria Guidelines 
• Each story must have 3–5 clear, testable acceptance criteria 
• Criteria must follow Given / When / Then (GWT) format where possible 
• Stories must be independent, negotiable, valuable, estimable, small, and testable (INVEST) 
• All stories must be linked to at least one JUnit test case 
 
4.2 Sample User Story Template 
FIELD CONTENT 
Story ID US-001 
Title User Registration 
As a... New visitor to the platform 
I want to... Register with my email and password 
So that... I can access personalized features 
Acceptance Criteria 
1. Given valid credentials, when I submit the form, then I receive a confirmation 
email. 2. Given an existing email, when I register, then the system shows an 
error. 3. Given missing fields, when I submit, then validation errors highlight 
each field. 
Priority High

6. Code Repository & CVS Guidelines 
Every team must maintain a single Git repository for the entire project. Follow the branching and commit 
conventions below to ensure a clean, reviewable history for evaluation. 
 
6.1 Repository Setup 
1. Create one shared repository per team on GitHub, GitLab, or Bitbucket 
2. Name the repo using the format:  gl-spark-<team-name>  (e.g., gl-spark-phoenix) 
3. Add all team members as collaborators with write access 
4. Add the GL evaluator account as a read-only collaborator before 2 April EOD 
5. Initialize the repo with a README.md describing the project 
 
6.2 Branching Strategy 
BRANCH PURPOSE WHO MERGES 
main Production-ready, stable code only Team Lead (PR + review) 
develop Integration branch for all features Team Lead 
feature/<story-id> One branch per user story Developer (PR to develop) 
hotfix/<issue> Critical bug fixes only Team Lead 
 
RULE: Never commit directly to main. All merges must go through a Pull Request reviewed by at least one 
other team member. 
 
6.3 Commit Message Convention 
Follow the Conventional Commits standard for all commit messages: 
 
PREFIX MEANING EXAMPLE 
feat: New feature or user story feat(US-001): add user registration 
endpoint 
fix: Bug fix fix(auth): resolve JWT expiry issue 
test: Add or update tests test(US-002): add JUnit tests for order 
service 
docs: Documentation updates docs: update README with setup 
steps 
chore: Dependency / config changes chore: update Spring Boot to 3.2.1 
refactor: Code restructure (no logic change) refactor: extract OrderValidator class

6.4 Recommended Repository Structure – As per best Practices  
6.5 Code Quality Standards 
• All REST APIs must follow standard HTTP methods (GET, POST, PUT, DELETE) with proper status 
codes 
• Each service must have a minimum of 4 JUnit 5 unit tests using Mockito for mocking 
• Code must follow TDD: write tests first, then implement to pass them 
• No dead code, no System.out.println in production code — use SLF4J / Logback 
• All public methods and classes must have Javadoc comments 
• Follow Google Java Style Guide for formatting 
 
6.6 Pull Request Checklist 
Before raising a PR to develop, every developer must verify: 
6. Code compiles without errors ( mvn clean compile ) 
7. All unit tests pass ( mvn test ) 
8. No merge conflicts with the target branch 
9. PR description links to the User Story ID (e.g., Closes US-003) 
10. At least one reviewer is assigned 
11. No API keys, passwords, or secrets committed — use environment variables

8. Team Checklist — Before Evaluation 
Use this checklist daily. All items must be checked by 9 April EOD. 
 
8.1 Documentation & Planning 
• PRD completed with Use Case and Functional Requirements 
• Minimum 4 User Stories with Acceptance Criteria documented 
• All 7 diagram types created and saved as slides (PPT/PDF) 
 
8.2 Architecture & Code 
• Minimum 2 independent microservices implemented 
• API Gateway configured and routing all service calls 
• Each service has REST endpoints following HTTP standards 
• Frontend (Angular/React) connected to backend services 
• Service Registry / Discovery configured (good-to-have) 
• Resilience patterns implemented (good-to-have) 
 
8.3 Testing 
• TDD approach followed — tests written before implementation 
• Minimum 4 JUnit 5 unit tests per service 
• Mockito used for dependency mocking in tests 
• All tests passing (mvn test → BUILD SUCCESS) 
 
8.4 Repository 
• Git repo created and shared with evaluator 
• Branching strategy followed (main / develop / feature branches) 
• Conventional commit messages used throughout 
• README.md with setup and run instructions 
• No secrets or API keys committed to the repo 
• All PRs reviewed and merged before code freeze
