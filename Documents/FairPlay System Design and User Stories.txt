 
📄 FairPlay – Complete System Design & 
User Stories Document 
 
1. Project Overview 
FairPlay is a sports venue and activity booking platform that enables: 
● Users to discover, join, and host sports activities 
● Users to book sports venues 
● Venue owners to manage venues and bookings 
The system follows a microservices architecture using: 
● Spring Boot 
● Eureka (Service Discovery) 
● OpenFeign (Inter-service communication) 
● Angular (Frontend) 
 
2. System Architecture 
🔹 Components 
● API Gateway 
○ Single entry point 
○ Routes requests to services 
● Eureka Server 
○ Service registry & discovery 
● User Service 
○ Handles users & activities 
● Venue Service 
○ Handles venues & bookings 
 
🔹 Flow 
1. Client → API Gateway 
2. Gateway → Routes to service 
3. Services register with Eureka 
4. Venue Service calls User Service via OpenFeign

4. User Stories 
(Optimized for microservices) 
 
🟢 US-001: User Registration & Login 
As a user 
I want to register and log in 
So that I can access the platform 
Acceptance Criteria: 
● Given valid data → user is created 
● Given duplicate email → error 
● Given valid login → success 
 
🟢 US-002: Profile Management 
As a user 
I want to update profile 
Acceptance Criteria: 
● Valid update → saved 
● Invalid input → error 
 
🟢 US-003: Host Activity 
As a user 
I want to host activity 
Acceptance Criteria: 
● Activity created successfully 
● Missing fields → error 
 
🟢 US-004: Join Activity 
As a user 
I want to join activity 
Acceptance Criteria: 
● User added 
● Duplicate prevented 
 
🔵 US-005: Register & Manage Venue 
As a owner 
I want to manage venue 
Acceptance Criteria: 
● Venue created 
● Venue updated 
 
🔵 US-006: Search & Book Venue 
As a user 
I want to search & book 
Acceptance Criteria: 
● Search returns results 
● Booking confirmed if available 
 
🔵 US-007: Manage Bookings 
As a user 
I want to view/cancel bookings 
Acceptance Criteria: 
● Bookings fetched 
● Cancellation updates status 
 
🟣 US-008: User Validation (Feign) 
As a system 
I want to validate user 
Acceptance Criteria: 
● Valid user → proceed 
● Invalid user → fail

6. API Communication (OpenFeign) 
 
🔹 Why OpenFeign? 
Used for: 
● Service-to-service communication 
● Clean REST client abstraction 
 
🔹 Implementation 
In Venue Service: 
@FeignClient(name = "user-service") 
public interface UserClient { 
 
    @GetMapping("/users/{id}") 
None
    User getUserById(@PathVariable Long id); 
} 
 
🔹 Use Case 
Before booking: 
1. Venue Service calls User Service 
2. Checks if user exists 
3. If valid → booking allowed
